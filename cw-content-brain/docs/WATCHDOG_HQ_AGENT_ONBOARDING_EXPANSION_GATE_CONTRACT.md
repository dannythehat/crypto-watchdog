## 1. Purpose

Build 63 defines the mandatory onboarding gate every future Watchdog HQ agent, manager, validator, page builder, content builder, SEO worker, media planner, affiliate worker, analytics worker, or operational role must pass before being added to the system.

The purpose is to stop uncontrolled agent sprawl, blueprint drift, vague roles, duplicated responsibilities, unsafe permissions, and agents inventing their own rules.

Future agents must join Watchdog HQ like employees joining a real company: with a clear role, department, manager, permission boundary, input/output contract, workflow route, audit trail, escalation route, QA route, and safety lock.

## 2. Core Rule

No new agent, role, manager, validator, page-builder system, or department worker may be added unless it passes the Agent Onboarding & Expansion Gate.

A future agent must not be created from a vague instruction such as:

- build good pages
- improve SEO
- write content
- manage affiliates
- fix reviews
- create media
- analyse everything

Every agent must be specific, bounded, useful, auditable, and aligned with the locked CryptoWatchdog / Watchdog HQ vision.

## 5. Department Ownership Requirement

Every agent must belong to exactly one owning department.

The owning department is responsible for:

- task routing
- input quality
- output quality
- retries
- rejection loops
- escalation discipline
- manager sign-off
- avoiding duplicate work
- keeping the agent inside its lane

Agents may support other departments, but they must still have one primary owner.

## 6. Manager Ownership Requirement

Every agent must have a named manager.

The manager is responsible for:

- approving the agent's work type
- confirming the agent's boundaries
- receiving completed outputs
- handling routine issues
- escalating high-risk issues
- preventing the agent from bypassing workflow
- stopping duplicate or conflicting outputs

Specialist agents must not route directly to Danny unless the contract explicitly requires Danny escalation.

## 7. Vision Alignment Requirement

Every new agent must state how it supports the Watchdog HQ vision.

The agent must align with these principles:

- evidence first
- user protection first
- trust before affiliate revenue
- useful pages over thin SEO pages
- human and practical tone
- beautiful structured page experiences
- no generic AI content
- no fake testing claims
- no unsupported scam/fraud accusations
- no unsupported best/safest claims
- no random internal links
- no random affiliate placements
- no live site edits in current mode
- Safe Apply much later

## 8. Page-Type Awareness Requirement

Any agent that works on pages, articles, posts, reviews, hubs, comparisons, warnings, promos, or media must declare its page-type awareness.

The agent must know which content types it is allowed to work on.

Examples:

- review_page
- themed_hub_page
- guide_article
- blog_post
- news_post
- warning_page
- scam_risk_page
- comparison_page
- promo_offer_page
- tool_calculator_page
- glossary_page
- landing_page
- FAQ_page

An agent must not use the same structure for every content type.

## 9. Template Requirement

Any page/content agent must reference the correct template before producing output.

Template rules:

- review pages must follow the review template
- themed hub pages must follow the themed hub template
- guide articles must follow the guide template
- blog/news posts must follow the blog/news template
- warning/scam-risk pages must follow the serious-content template
- comparison pages must follow the comparison template
- promo/offer pages must follow the promo/offer template
- tool/calculator pages must follow the tool/calculator template

No future drafting agent may invent random article structures.

## 11. Validator Requirement

Every new agent must state which validator checks its work.

Validators may check:

- schema completeness
- template compliance
- evidence support
- unsupported claims
- affiliate disclosure
- internal link relevance
- external source relevance
- rating/trust-status safety
- serious-content tone
- media placement logic
- page-type relevance
- workflow state compliance
- audit event completeness
- blocked action compliance

An agent is not considered production-ready until a validator route exists.

## 12. Page Builder Specific Rule

A page builder is not a generic writer.

A page builder must understand:

- the page niche
- the page type
- the target user intent
- the required sections
- reviewed platform tiles
- comparison tables
- favourite/recommended sections where evidence supports them
- internal link opportunities
- external/source link requirements
- affiliate placement restrictions
- image placement
- video placement
- related blog/article placement
- related review placement
- related guide placement
- serious-content exceptions
- CryptoWatchdog final word
- QA and Danny approval route

Example: a Crypto Exchange Hub Page Builder must stay focused on crypto exchanges. It may reference related topics such as trading bots only where relevant and must route that reference to the correct guide/page. It must not turn the exchange page into a wallet, casino, tax, NFT, or trading bot page.

## 13. Beautiful Page Experience Requirement

Any page-builder or content-builder agent must support the long-term visual/content experience.

Where relevant, the agent must consider:

- hero section
- beautiful images
- useful diagrams
- reviewed platform tiles
- trusted brand boxes
- comparison tables
- side tiles
- in-section related content
- natural anchor links
- relevant videos
- FAQs
- CryptoWatchdog final word
- user journey from beginner explanation to decision support

The output remains report-only until future approved implementation systems exist.

## 14. Internal Linking Requirement

Internal links must be natural, relevant, and page-type aware.

Agents must avoid:

- random internal links
- keyword-stuffed anchors
- irrelevant cross-niche links
- over-linking
- linking to pages that do not support the section context
- linking to serious warning pages in a promotional way

Internal links should help users understand the topic and move to the right next page.

## 15. Affiliate and Commercial Requirement

Affiliate/commercial agents and page builders must obey:

- no affiliate URL insertion in current mode
- no commercial placement without disclosure logic
- no affiliate CTAs on serious warning/scam-loss pages unless manually approved
- no hiding commercial intent
- no ranking products based only on commission
- no “best” or “safest” claims without evidence
- no trust-rating edits
- Danny approval required for high-risk commercial decisions

## 16. Evidence and Trust Requirement

Agents must separate:

- verified evidence
- suspected issues
- public complaints
- Danny-provided evidence
- proof gaps
- unsupported claims
- opinion
- recommendation

Agents must not invent testing, screenshots, deposits, withdrawals, KYC checks, fees, support outcomes, user numbers, partnerships, licences, awards, ratings, or safety guarantees.

## 17. Workflow Integration Requirement

Allowed current states remain report-only states, such as:

- detected
- suspected
- verified
- recommended
- blocked
- monitor_only
- needs_more_evidence
- escalated_to_qc
- escalated_to_master_ai
- recommended_for_danny_review

Agents must not create active approved/applied states in current outputs.

## 19. Rollback and Incident Requirement

Every agent must have an incident route.

Incidents include:

- blueprint drift
- roadmap drift
- unsupported claims
- unsafe affiliate suggestion
- trust-rating risk
- serious-content tone failure
- invented evidence
- live edit attempt
- Supabase write attempt
- publishing attempt
- approval/apply bypass
- schema failure
- repeated validator failure
- wrong department routing
- duplicate ownership conflict

The rollback/incident route must follow the Build 62 contract.

## 20. Permission Boundary

Default permissions for every new agent:

- live publishing: blocked
- live site edits: blocked
- Supabase writes: blocked
- affiliate insertion: blocked
- trust-rating edits: blocked
- AI/API calls: blocked unless explicitly approved later
- media generation/upload: blocked unless explicitly approved later
- approval/apply workflow: blocked
- deleting/changing production data: blocked
- real content publishing: blocked

Any exception requires Danny approval and a future controlled build.

## 21. Blueprint and Roadmap Update Rule

The blueprint and roadmap must be updated when a new agent or role:

- creates a new department
- changes department ownership
- changes workflow states
- changes permission boundaries
- changes safety rules
- changes page/content templates
- changes validator requirements
- introduces a new page builder family
- introduces a new content type
- introduces a new escalation route
- marks a build complete
- changes the next planned build

This prevents blueprint drift.
