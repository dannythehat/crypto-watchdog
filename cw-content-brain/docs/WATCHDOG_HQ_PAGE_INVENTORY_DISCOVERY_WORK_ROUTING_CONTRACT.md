# Watchdog HQ Page Inventory, Discovery & Work Routing Contract v1

Build 68 locks the first hive-mind operating layer after the initial specialist page-builder contracts.

This is not a blank-page writing brief.

This is an operating contract for future Watchdog HQ agents, managers, validators, QA workers, and routing layers. It defines how the hive-mind discovers, inventories, classifies, prioritises, and routes CryptoWatchdog page work safely.

## 1. Current Mode

Current system mode: READ_ONLY_REPORT_ONLY.

Build 68 does not enable:

- live publishing
- live site edits
- Supabase writes
- affiliate URL insertion
- trust-rating edits
- AI/API calls
- media generation or download
- outreach
- email sending
- Safe Apply
- approval for publication
- deleting or changing production data
- real content publishing

All outputs remain local/report-only recommendations, routing objects, gap maps, and work queue items.

## 2. Purpose

Build 68 creates the page inventory, discovery, and routing control layer for Watchdog HQ.

Future agents will use this layer to identify existing pages that need refresh, thin pages, outdated pages, missing evidence, SEO gaps, internal-link gaps, missing related content, missing page opportunities, missing hub opportunities, duplicate or cannibalisation risks, affiliate disclosure issues, and sensitive warning/scam-risk pages requiring QC or Danny escalation.

The purpose is to route work to the correct specialist team without inventing new rules, bypassing managers, weakening evidence discipline, or drifting away from the blueprint.

## 3. Allowed Page Inventory Sources

Future inventory and discovery work may use local/report-only sources only.

Allowed current sources:

- local sitemap data where available
- local snapshot/export data where available
- existing cw-content-brain reports
- local content inventories
- local metadata reports
- local page quality reports
- local internal link reports
- local content cluster and related section reports
- local analytics/search imports where already provided by Danny
- existing page-builder contracts
- Watchdog HQ blueprint documents
- Watchdog HQ roadmap documents
- existing report-only dashboard exports

Blocked current sources:

- live crawling unless explicitly approved in a future controlled build
- live fetching unless explicitly approved in a future controlled build
- Supabase writes
- production database edits
- live website edits
- unpublished external API calls
- AI/API calls
- media downloading
- media generation
- outreach or email sending

## 4. Discovery Categories

Every finding must be classified using one or more of the following discovery categories:

- existing_page_needs_refresh
- existing_page_thin_or_weak
- existing_page_outdated_structure
- existing_page_missing_evidence
- existing_page_poor_seo_structure
- existing_page_internal_link_gap
- existing_page_missing_related_content
- existing_page_needs_media_planning
- missing_page_opportunity
- missing_hub_opportunity
- duplicate_or_cannibalisation_risk
- affiliate_or_commercial_opportunity
- warning_or_sensitive_page_risk
- serious_content_qc_required
- monitor_only

Agents must not create vague recommendations. Every item needs a category, page or proposed page target, reason, owner, next state, and safety status.

## 5. Existing-Page-First Rule

If a CryptoWatchdog page already exists, future agents must audit, refresh, rebuild, enrich, link, restructure, or route the existing page before proposing a duplicate page.

Duplicate page proposals are blocked unless the agent clearly explains why the existing page cannot serve the user or search purpose, why a separate page is needed, what template applies, what evidence is required, what internal links are needed, whether cannibalisation risk exists, and whether Danny review is required.

## 6. Missing-Page Rule

Missing-page opportunities are allowed as recommendations only.

A missing-page recommendation must include why the page appears missing, what user or search purpose it serves, which template or hub contract applies, what evidence or source support is needed, which department owns it, which manager owns it, whether Gatekeeper Grace review is required, whether Danny review is required, and why the recommendation remains report-only.

## 7. Routing Logic

Findings must be routed through the Watchdog HQ management structure.

| Discovery area | Owner department | Manager / agent route |
|---|---|---|
| Page structure, page type, hub structure | Editorial Strategy | Blueprint Bella |
| Search intent, metadata, cannibalisation, keyword gaps | SEO | Rankhound / Keyword Kev |
| Internal linking gaps | Internal Linking | Linksmith |
| Routing, queue control, work ownership | Content Operations | Routey Rachel |
| Future rebuild candidates | Content Production | Rewrite Rita, only after safety gates |
| Evidence gaps and proof issues | Evidence | Inspector Proof |
| Images, screenshots, video placement planning | Media | Image Iris / Storyboard Sam |
| Affiliate placement and disclosure risk | Affiliates / Offers | Disclosure Daisy / Offer Owl |
| Unsupported claims, scam wording, trust risk, serious content | QA / Governance | Gatekeeper Grace |
| Traceability and event logging | Audit / Governance | Audit Alfie |
| Prioritisation and Danny-ready filtering | Command | The Gaffer |
| CEO-level exceptions and critical approvals | Human Owner | Danny |

Danny receives CEO-level exceptions, not routine operational noise.

Danny-only decisions include trust/rating changes, scam/fraud/legal wording, affiliate-sensitive decisions, publication-sensitive decisions, strategic brand decisions, override requests, unresolved governance conflicts, and anything Gatekeeper Grace escalates as critical.

## 8. Priority Model

Allowed priority values:

- critical
- high
- medium
- low
- monitor_only

Critical items involve serious scam/fraud/legal wording risk, trust/rating change risk, user safety risk, victim-help or financial harm content, affiliate conflict, unsupported claims that could mislead users, blueprint drift, roadmap drift, repeated failure loops, or governance incidents.

High items involve important existing page weakness, major SEO opportunity, high-value hub gap, major internal-link gap, evidence review need, or commercial opportunity requiring disclosure review.

Medium items improve quality, structure, links, UX, or SEO but are not urgent or high-risk.

Low items are useful but not time-sensitive.

Monitor-only items are possible issues without enough evidence to recommend action.

## 10. Allowed Workflow States

Allowed current states include:

- detected
- suspected
- verified
- recommended
- blocked
- monitor_only
- needs_more_evidence
- escalated_to_qc
- escalated_to_master_ai
- recommended_for_danny_review

Blocked current states:

- approved
- applied

Approved and applied are future-only operational states. They must not appear in current work queue outputs as active states.

## 11. QA and Safety Gates

Gatekeeper Grace must receive any item involving unsupported claims, trust/rating changes, scam/fraud wording, legal or regulatory wording, affiliate conflict, serious harm, safety, victim, warning content, invented testing, invented proof, invented screenshots, misleading claims, publication-sensitive recommendations, or serious-content tone issues.

QA never edits. QA passes, rejects, blocks, or escalates.

## 12. Audit Trail Requirements

Every routed item must be traceable through Audit Alfie using Build 61 audit/event contract principles.

Future work items should preserve trace_id, source report, source page or path, detecting agent, routing agent, owner department, owner agent, workflow state before and after routing, safety checks, permission checks, rejection or escalation reason, Danny review requirement, and blocked action confirmations.

## 13. Relationship To Page-Builder Contracts

Build 68 does not replace specialist page-builder contracts.

It sits above them as a discovery and routing layer.

When Build 68 finds a page issue, it routes the work to the right page-builder, SEO, evidence, media, affiliate, QA, or content production contract.

## 14. Acceptance Criteria

Build 68 is acceptable only if it strengthens the hive-mind operating system, preserves READ_ONLY_REPORT_ONLY, supports existing-page-first behaviour, avoids blank-page content generation, routes findings to departments and managers, protects Danny from operational noise, escalates CEO-level exceptions correctly, keeps approved/applied blocked, preserves evidence-first discipline, protects affiliate disclosure and trust, protects scam/fraud/legal wording, supports audit traceability, and updates the blueprint and roadmap.

## 15. Build 68 Safety Confirmation

Build 68 does not publish content, edit live pages, write to Supabase, insert affiliate URLs, change trust ratings, call AI/API tools, generate media, download media, send outreach, send emails, approve publication, activate Safe Apply, or delete/change production data.

Current mode remains READ_ONLY_REPORT_ONLY.
